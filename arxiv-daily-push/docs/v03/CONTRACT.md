# R0-8 工作合同（一页）

> 每次开发会话置顶此页。这是唯一的执行仪式；除 CHANGELOG 一行与 STATUS 一行外，禁止新增任何治理文书。

## 追寻目标（一句话）

在不启用任何生产副作用的前提下，用最少的新代码打通「抓取 → 排序 → 讲义 → 主动回忆 → 间隔复习」这一条单用户真实闭环；每一步产出可追溯证据，能复用成熟开源项目的绝不自研；任何不直接服务本条闭环的新文件、新文档、新测试一律视为超范围；每周五必须交出 Owner 能亲手操作的可用版本，同一问题修复不得超过两次。

## 硬约束

1. 一个任务 = 一次会话干完：改动 ≤5 文件、上下文 ≤100 KB、有一条可运行的验收断言。
2. 复用优先：py-fsrs / feedparser / pyalex / semanticscholar / FastAPI+Jinja2+htmx / launchd / SQLite FTS5。写任何算法前先回答：哪个成熟项目已解决它？为什么不能用？
3. 参数只读 `config/thresholds_v0_3.yaml`；不新造权重、公式、状态或版本域。
4. 不变量不可破坏（02_系统架构/业务规则与系统不变量.md 十条）；无授权凭证则副作用失败关闭；同幂等键永不二发；每次 run 落一行 manifest（正常/降级/弃权/失败/未运行）。
5. 完成定义：验收断言绿色 → `docs/v03/04_开发记录/CHANGELOG.md` 追加一行 → `docs/v03/STATUS.yaml` 更新对应阶段。
6. 遇到需要 Owner 决策的分叉：停下，列出选项，不代替 Owner 选择。

## 修复熔断

同一问题修复 ≤2 次、每次 ≤半天；问题台账每问题最多两行。第 3 次触发换方案，按优先级：回滚上一可用版 → 降级砍掉该功能照常交付 → 换成熟开源组件。周五到点必须交付：可以带「已降级功能清单」，不可以带「正在修，下周一定好」。
