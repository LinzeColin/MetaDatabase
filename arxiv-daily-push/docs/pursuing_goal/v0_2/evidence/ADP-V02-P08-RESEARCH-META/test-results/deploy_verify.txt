（`cn_run_log` 有 `2026-07-17T0414-manual` 的成功记录），故 `POST /api/run` 返回「未运行：当日已成功运行，幂等跳过」。

**我没有去强行绕过这个幂等守卫。** 绕过它要删掉当天的 `cn_run_log` 行，那会让 `ingestAll` 与
`selectDaily` 一并重跑 —— 可能**改掉 Owner 今天已经发布的当日精选**。为了我自己验证方便去动
线上已发布的内容，不划算，也违反「不得改写既有生产数据」。

因此以下三项**在下一次 cron 之前仍是未验证的**（复核第 6 轮的残余风险 #4，原样保留）：
1. `env.DB.batch([q1, q2])` 读 SELECT 结果 —— 这是**全仓唯一**一处用 batch() 跑 SELECT 的地方，
   正确性依据是 Cloudflare 文档的 `D1Result[]` 契约，**没有仓内先例**。若该契约与文档不符，
   `results` 会为空 → enrichMeta 静默地什么都不补（不会报错、不会降级、页面照常）——
   这正是本项目最怕的「看着绿、其实什么都没做」，故**必须在首次 cron 后实查**。
2. 真实 D1 的查询计划（本地用的是 sqlite 3.43.2 + 真 schema 作代理）。
3. 真实 cron 的 CPU 余量。
