# Known gaps · ADP-V02-P03-SEARCH（诚实边界）

本次只交付 T060 中**线上数据真能支撑**的部分。以下是**明确未做/做不到**的，页面文案也不作相应声称。

## 1. 不是 T060 的 O(1) 精确查找，只是「对已召回行的置顶」
T060 的 `exact_lookup` 是建在**全部文档**上的哈希索引，承诺"精确标识符**每次**都是第一条"。
线上没有 `doc_number`/`doi` 列，worker 只能对 SQL `LIKE` **已召回的 ≤40 行**做重排。
→ **能力严格弱于 T060**；页面因此只说"若查询出现在某条的文号／DOI 这类标识符里，该条会被置顶"，
不再承诺"精确命中会排在最前"（该无条件承诺已被独立复核判定为过度声称并移除）。

## 2. 文号的空格变体端到端不成立（`normIdent` 只影响排序，不影响召回）
存的是 `国发〔2026〕12号`；用户键入 `国发 〔2026〕 12号` 时 SQL `LIKE` **返回 0 行**，重排器根本不会运行。
复核实测：`"国发〔2026〕12号"→found=1`；`"国发 〔2026〕 12号"→found=0`；`"国发〔2026〕12 号"→found=0`。
真正的归一化精确查找需要**落库的归一化标识符列 + 回填**（新列/新表），本次刻意不做，也**不在页面上声称**。
（`normIdent` 的 ASCII 大小写折叠在召回层其实冗余——SQLite `LIKE` 本就 ASCII 大小写不敏感。）

## 3. ★「标识符匹配」只证明标识符文本匹配，不证明该行就是官方原件★
线上 board3 的数据源**全是媒体新闻 RSS**（人民网／中新网／新浪）——行是**关于**某文件的报道／解读，
**不是文件本身**。因此徽章只敢叫「标识符匹配」（可证：查询出现在该行抽出的文号/DOI 里），
**不敢叫「精确命中」**（不可证：库里可能根本没有那份原件）。
→ 只有 **P04（board3 接 A0 官方原文入每日流水线）**落地后，库里才会有原件，这个语义才可能升级。

## 4. agency / region / status 三个 facet 未做
T060 的 `structured_filter` 含 agency/region/status；`cn_items` **没有这些列**，
要做必须先抽取并落库（新列/新表 + 回填）。本次只做**线上索引真能支撑**的 facet：
**板块（`idx_cn_items_board_recency`）+ 日期范围（`idx_cn_items_recency`）**。

## 5. 召回上限 40 / 无筛选时仍是扫描（既有行为，未因本次变差）
- 只对**最近 40 条**命中做重排；若精确的那条排在 40 名之外，不会被置顶。页面明示"仅显示最近 40"。
- 无 board 筛选时仍为 `SCAN cn_items USING INDEX idx_cn_items_recency` —— 这是**改动前就存在**的行为，
  本次**未使其变差**（复核用 EXPLAIN QUERY PLAN 逐一比对，计划与改前逐字节一致）；
  给出 board 筛选时反而**收窄**为 `SEARCH ... USING INDEX idx_cn_items_board_recency (board_id=?)`，是净改善。
- `LIKE %x%` 天然不可走索引，故检索面加上 `id`/`url` 两列**不改变查询计划**（复核实测同计划）。

## 6. 抽取器的贪婪前缀
`FS_DOCNUM_RE` 的 `[一-龥A-Za-z]{0,8}` 会吞入前文（「转发国发〔2026〕12号」→ 抽出「转发国发〔2026〕12号」）。
本次用 `includes` 而非 `===` 容忍该问题（否则真文件会被漏掉），但**未改正则**——
改它会同时改变 P01 已上线的 factsheet 行为并偏离 `extract_factsheet.py`，应作为单独任务评估。
