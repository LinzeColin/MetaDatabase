# Known gaps · ADP-V02-P09 — 覆盖与缺口上线（T043）

## 1. ★为什么没有照搬 v0_1/tools/gap_detector.py★
拿【发货的那个工具】对【真实线上数据】跑（847 条 / 31 源 / 127 月 = 3937 格）：

    unexplained: 0                    -> T043 验收 PASS
    source_not_yet_active: 3665 格    -> 93%

`infer_source_windows` 从【我们自己抓到的条目】反推源的起始月。我们最近才开抓，于是
arxiv-all 的 active_from 被推成 2026-07，2016-01..2026-06 全被标成「那时这个源还不存在」。
**对 arXiv（1991）、Nature（1869）、NEJM（1812）、PNAS（1915）、bioRxiv（2013）这就是假的**
——光这 8 个源就有 **1003 格**假解释（复核独立复现，并指出该选择还刻意排除了 nature-machine-intel，
因为它 2019 年才创刊，那 36 格是**真的**史前空白）。

**即：T043 的验收是靠一个假解释通过的**，而它**分辨不出「诚实的空洞」和「错的解释」**。
错的解释比空洞更糟。故本次上线**不引入 `source_not_yet_active` 这个判定**：
没有独立证据证明某源当时不存在，就不许这么说。默认 `not_backfilled` 是**可证的事实**。

## 2. ★同一个谬误，我在另一根轴上又犯了一遍（复核 BLOCK-1）★
第一版把源清单从 `cn_items` 推出来 → **一条都没抓到的源整个从网格里消失**：
线上 `cn_sources`=37、有条目的仅 31 → **6 源 × 127 月 = 762 格既无 count 也无解释**，
**当场违反 T043 的「0 个静默未解释空洞」**。

> 复核的原话最该记住：**v0_1 从条目反推「源的起始月」→ 没数据＝「那时源还不存在」；
> P09 从条目反推「源的清单」→ 没数据＝「这个源不存在」。同一个谬误，从时间轴挪到了源轴。**

而「一条都没抓到的源」**恰恰是缺口检测存在的意义**。线上那 6 个是：
`cell、cell-neuron、lancet、science-advances、stats-gov、gnews-us-tech`
——**其中 `stats-gov` 是 P04 我自己接进来的 A0 官方源，至今一条都没抓到**。旧网格会永远藏着它。
现改为由 `cn_sources` 驱动（`radarPage` 30 行外就是这么做的，我没看），并把这 6 个源**点名**印在页面上。

## 3. 覆盖率也因此被抬高过（复核 BLOCK-2）
分母漏掉那 762 格 → 1.8%；诚实分母（37 × 127 = 4699）→ **1.5%**。
这个数字**就是本批的全部论点**，抬高它等于自毁。已随 §2 一并修好。线上现值：**69 / 4699 = 1.5%**。

## 4. 「0 未解释」不再被导出（复核 BLOCK-4）
原先导出 `unexplained: 0` 并断言它 —— 那是**恒真字面量**，任何改动都推翻不了，
而这份套件上面刚吹过自己删掉了一条 `ok(...||true)`。**同一个毛病**。
现在不导出它，改看 `coverage_pct`（债务）与 `sources_in_registry` / `sources_with_items` 的差。

## 5. 验收套件曾经是哑 mock（复核 BLOCK-3）
`prepare(q)` 不看 `q` → 把发货查询换成 `SELECT this_column_does_not_exist FROM table_that_does_not_exist`
仍然 `ACCEPTANCE = PASS`。**一条每次 /system 都会 500 的查询能干净通过验收。**
这与 P08 第 1 轮 BLOCK 的 F1 是同一个错 —— 我在 P08 用真 sqlite 修好了，**转头在 P09 又写了个哑 mock**。
现用真 `node:sqlite` + 真 `schema_cloud.sql`，nc7 钉死。

## 6. DIR-007（实测，rows_read 才是 D1 计量的量）
- 聚合查询：**rows_read = 1694**（≈ 2 × 表大小 847；全表扫 + `USE TEMP B-TREE FOR GROUP BY`），结果行 69
- 登记查询：**rows_read = 74**，结果行 37
- 合计约 **1768 行 / 每次 /system 访问** → D1 免费档 5M 行/天 ≈ **2800 次访问/天**
- CPU：网格 37×127 在 JS 里算，实测 0.51ms（10ms 预算）
- ★两句诚实话★：
  1. 结果行 69 **不是成本**，`rows_read` 才是。原注释「行数 = 不同 (source,month) 组合数」说的是**结果行**，
     读的人会当成开销 —— 那是误导（复核指出，且 `rows_read` 此前**从未被测过**）。
  2. 这是**全表扫、随 `cn_items` 线性增长**：`substr(COALESCE(...))` 不可索引，GROUP BY 全表本就要扫完。
     今天 `KEEP_PER_BOARD=300` 把表压在约 847 行故无压力；**若将来真做 2016+ 回填把表撑大，
     每次 /system 访问都会跟着涨** —— 那时必须改成物化/缓存。写在这里，别等它咬人。
