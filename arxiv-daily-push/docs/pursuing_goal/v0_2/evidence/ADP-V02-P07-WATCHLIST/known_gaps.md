# Known gaps · ADP-V02-P07 — 关注 /watchlist

## 1. ★这不是 T066，是它在线上数据能支撑的诚实子集★
T066 (`tools/watchlist_digest.py`) 的规格要求两样**线上并不存在**的东西：

| T066 要求 | 线上现实 |
|---|---|
| `matches()` 在 **`topic/agency/region/entity/doc_number`** facet 上精确匹配 | `cn_items` 只有 board/source/kind/title/url/summary/categories/authors/published_at ——**这些 facet 字段一个都没有** |
| `run_digest()` 用 **T026 `content_hash`** 判「实质变化」，并按 `canonical_id` 归并 | **S2 版本层未上线**；没有 `canonical_id`，也没有 `content_hash` |

因此本次**只交付线上真能支撑的**：
- **`doc_number`** —— 用 P01 的确定性抽取器从 title/summary **真实抽出**后匹配（board3 的 A0 官方原文标题常带文号，如 `发改环资〔2026〕1062号`）。与 P03「标识符匹配」同口径（容忍抽取吞入的前缀，故「转发国发〔2026〕12号」这类真文件不会漏）。
- **`board`** —— `cn_items.board_id` 真字段，**精确**匹配（这一条是忠实于 T066 的）。
- **`keyword`** —— 标题/摘要子串。**★这不是 T066 的 facet，是 ADP-V02 追加★**（因为只有 doc_number+board 太薄）。
- **未交付**：`topic` / `agency` / `region` / `entity` —— 需要 S5 的实体解析（未上线）。**不臆造这些字段**。

## 2. ★变化判定只做「新条目」，不做「内容实质变化」★
T066 的「实质变化」= 新的 canonical 条目 **或** `content_hash` 变了（模板噪声重渲**不算**变化）。
没有 T026 版本层，**已入库条目的内容改动无法检测**。故本实现只报**新条目**，页面也**明说**「只检测新条目……故不做也不声称」。
**保留的 T066 性质**：可重入 —— ack 记录 `(watch_id,item_id)`，再看即**零未读**，不重复打扰（实测）。
**未做**：T066 的 **silence signal**（被关注源在周期内无变化时的静默信号）——它依赖周期与 source 维度的聚合，本实现是即时视图，无周期概念。

## 3. ★复核 BLOCK 抓到的真实缺陷：我的有界性分析只算了单条★
复核判定（实测，非推测）：
- 我把**循环写反了**。T066 是 `for it in items: for w in watches:`（**一遍**）；我写成**每条关注各扫一遍**（查询 + 正则）→ **×20**。
- `FS_DOCNUM_RE` 在**中文正文**上约 **0.041ms/行**（`[一-龥A-Za-z]{0,8}` 让 V8 在每个汉字位置都无法跳过；英文约 **0.05ms/1000 行**，差约 800 倍）。
- 实测 100 中文/900 英文 = **4.17ms/条 → ×20 = 83ms**；250 中文 = 207ms。而 **Workers 免费档 CPU 限额 = 10ms/次**（仓库自证：`idempotency_harness.py:85` `"plan": "Free (Cron Triggers + D1 + Workers free tier)"`）→ **约 3 条文号关注就打爆**，而我页面上写着上限 20。
- 更糟：**无年份文号**（`第12号令` —— `FS_DOCNUM_RE` 自己的第二分支就支持，我的 `/api/watch/add` 还 303 收下了）会让粗筛退化成 `LIKE '%%'`，1000 行全打进正则；而带年份时 `'%2026%'` 又**优先选中恰恰最贵的中文行**。
- 这与 P01-FACTSHEET 被 BLOCK 的**是同一类**（「跑在请求路径会打爆 Worker CPU」），且**用行数放大抵消了 P01 自己的 `slice()` 先例**。

**修复（按复核指出的根因，不是打补丁）**：
- **恢复 T066 的循环序**：`watchDigest()` **一次**共享窗口扫描 → `for (const it of rows) for (const w of watches)`。删除 `watchCandidates`/`watchUnseen`（0 残留引用）。
- **每条目只抽一次文号**，且只对 `title+summary` 含 **`号`** 的行跑正则。
- **实测（复核的语料与方法，best-of-9）**：100 中文/900 英文 ×20 条 **83ms → 0.92ms**；250 中文 **207ms → 1.64ms**；最坏（无年份 `第12号令` ×20 @250 中文）**1.53ms**；命中数不变。

**`号` 预筛的可靠性已证明（不是想当然）**：`FS_DOCNUM_RE` 两个分支**都强制**含字面 `号`；20 万随机串模糊测试中匹配 132 次、**不含 `号` 的 0 次**（复核独立复跑 40 万串、280,494 次匹配，同样 **0** 反例）；四种真实形态全含 `号`；英文语料完全没有 `号`。故预筛**不可能丢弃真命中**——纯加速、不改语义。
（复核另确认：预筛测的是 `title+summary` 拼接串，而 `fsFirst` 是逐字段扫描——拼接是字符超集，故不会有假阴性。）

## 3b. ★第二次 BLOCK：我修的是症状，不是原理★
复核在复验时**再次 BLOCK**，因为我**只修了 `doc_number`，把一模一样的 bug 留在了 `keyword`**：
`watchMatches` 里那行 `hay = (title+' '+summary).toLowerCase()` 是**只与条目有关**的工作，却写在 **per-(item, watch)** 的内层 → 1000×20 = **2 万次重建**。我把 `docs` 提出去了，**却没提 `hay`**。**×WATCH_MAX 的乘数只是被我挪了个位置，根本没消失。**

复核的原话是这一整轮最该记住的一句：
> **「Restoring T066's loop order doesn't fix anything by itself; hoisting item-only work out of the inner loop is what fixes it.」**

实测（复核）：20 条 keyword = **29.0ms**（全中文）／**7.95ms**（**纯英文也吃掉 80% 预算**）；**约 7 条就越过 10ms**；且成本**严格线性**于关注数（1/2/5/10/20 → 1.41/2.80/6.98/14.10/28.68）——线性本身就是"每一对都在做活"的铁证。

**按原理修**：内层循环里**不得有任何**「只与条目有关」或「只与关注有关」的计算。
- `hay` → **每条目算一次**；`w._nv`（doc_number 用 `normIdent`，其余用 `toLowerCase`）→ **每条关注算一次**；`watchMatches` 现在**零** per-pair 计算。
- 实测：全中文 20 条 **29.0 → 1.36ms**；纯英文 **7.95 → 0.93ms**；7 条临界点 **10.02 → 0.65ms**；混合 10+10 = 2.26ms。
- **乘数消失的证明**：关注数 1/2/5/10/20 → **0.38/0.37/0.64/0.98/1.31ms**（×20 关注只 ×3.4 成本，而非 ×20）。
- 正确性无损：keyword 指南=1000、board=1000、doc `国发〔2026〕7号`=**精确 1 条**、大小写不敏感、`%` 为字面量、已读正确排除。

## 3d. ★第三次 BLOCK：我的修复 #3 亲手废掉了我的修复 #2★
我为「已读集可能被 LIMIT 截断」加了 `ORDER BY at DESC LIMIT 5000`，理由是「窗口 ≤ `WATCH_SCAN` 条，故最近的若干次已读必然覆盖窗口」。
**这个前提是错的**：**`WATCH_SCAN` 限的是读取量，不是窗口**。30 天窗口在 5 万语料里有 **3875 条**、线上 20 万条时约 **1.5 万条** —— 已读集会超过**任何** N。
一旦截断，被截掉的是**最早 ack** 的行。这在「ack 时序 ≈ 发布时序」时无害（所以我的测试全过），但恰好在两者**背离**时咬人：
**未来日期的 `published_at`（政策生效日）永远排在候选第 1 位，它的 ack 时间却会老化到 5000 之外 → 重复通知** —— 正是修复 #2 存在的那个洞。
> 复核原话：**「You closed the hole in the write path and re-opened it in the read path. Fix #2 keeps the row; fix #3 refuses to read it.」**

复核的证据是构造性的：**6001 行已读 → 重复通知 YES；400 行（LIMIT 不生效）→ no**。所以**是 LIMIT 造成的**，与语料无关。
且我的 `ORDER BY` 还让该查询**丢了覆盖索引**（`SEARCH ... USING COVERING INDEX` → `SEARCH + USE TEMP B-TREE`，慢约 2.6×、每页 ×20）；若将来真有 `ANALYZE` 统计，还会退化成全表 `SCAN`（潜伏陷阱）。

**修复**：**把 `ORDER BY` 与 `LIMIT` 一起删掉**（`WATCH_SEEN_CAP` 随之删除，0 残留）。
- 截断变成**构造上不可能**，而不是「概率上不太会」。
- **有界性交回给「已被复核验证正确」的按窗口剪枝**——不能靠一个会**静默牺牲正确性**的 LIMIT 来换有界。
- 计划回到 `SEARCH cn_watch_seen USING COVERING INDEX (watch_id=?)`（索引内读取）。
- **复现复核的反例**：6001 行已读（全在窗口内、剪枝不删）+ 未来日期政策（ack 最早）→ **重复通知 = NO**，未读 999 正确排除该条。

**未采纳复核建议的「用候选 id 反向约束已读读取」的理由（诚实说明）**：要么需要约 1000 个 IN 绑定参数（触碰 SQLite 变量上限），要么用相关子查询——但后者仍会扫完该关注的整个已读集，**并不真正减少读取行数**。若此推断有误，欢迎推翻。

## 4. DIR-007 有界性（两层，都是实测换来的）
1. **行数**：候选查询必须「**时间窗 + LIMIT**」双重有界。只用 `LIKE + LIMIT` 时，**匹配不到**会为了凑满 LIMIT **走完整个索引**（真实 schema 5 万行实测 **25.0ms**；线上约 20 万条 × 20 条关注 = 单次访问读上千万行 → **一次就烧穿 D1 每天 5M 免费额度**）。加时间窗后计划变为 `SEARCH cn_items USING INDEX idx_cn_items_recency (<expr>>?)`（**索引范围定位**），无匹配也仅 **1.51ms**。
2. **CPU**：见 §3。
**实际读取量（不再引用已删除的 `WATCH_SEEN_CAP`）——三种证据强度分开标注**：
- **实测**：5 万条语料下 30 天窗口内 = **3,875 条**；查询计划、CPU 与反例复现均为实测。
- **外推**：线上约 20 万条时窗口内**约 1.5 万条**（按同比例外推，**非实测**）。
- **推导**：下表由「匹配率 × 窗口内条数」**算术推导**，非逐格实测。
- **候选**：**一次**共享扫描，**≤ `WATCH_SCAN`(1000) 行**（时间窗 + LIMIT，见上）。
- **已读集**：每条关注一次**覆盖索引**读取，**无 LIMIT** —— 其大小 ≈ **匹配率 × 窗口内条数**（线上窗口约 1.5 万条），**不是** 5000，也**不**被 `WATCH_SCAN` 限制（`WATCH_SCAN` 限的是候选读取量）。

| 关注类型 | 已读/条 | ×20 + 候选扫描 | 免费档 5M/天 可支撑访问数 |
|---|---|---|---|
| 窄关键词（匹配率 1%） | 150 | ~4k | ~1250 |
| 板块（按均分 25%） | 3,750 | ~76k | ~65 |
| 关键词 50% | 7,500 | ~151k | ~33 |
| **极宽关键词（100%）** | **15,000** | **~301k** | **~16** |

（**「均分 25%」只是记账口径，不是事实**：board1 是 arxiv 全站 + bioRxiv 的论文流，board3 是政策原文——**board1 几乎必然远高于 25%**，故真实的板块关注会落在该行与 100% 行之间；100% 行给出了上界，故信封仍然诚实。）

**诚实结论**：典型混合关注（平均约 15%）≈ **46k 行/次访问 ≈ 108 次/天**，对单用户够用；但**尾部风险是真的**——**已读读取现在是候选扫描的约 300 倍**，按本文件 §4 自己的标准，**它才是主导项**。它**只烧配额，不破坏正确性**（覆盖索引、纯索引内读取）。**若将来关注变多/变宽，这里是第一个要优化的地方。**
（复核建议过「用候选集成员资格剪枝」把已读压到 ≤1000/条，但**复核自己验证后否决了**：`worker_cloud.js:239` 的 `ON CONFLICT(id) DO UPDATE SET fetched_at=excluded.fetched_at` 会在每次重抓时抬高 `fetched_at`，而 `published_at` 为空的条目其 recency 就是 `fetched_at` —— 条目可以**跌出前 1000 后又跳回第 1 位**；若其已读行已按候选集被剪掉，就会**重复通知**，与我们刚堵的是同一类洞。故**不采纳**，按窗口剪枝才是正确谓词。）

## 5. 时间窗的语义代价（诚实）
`WATCH_WINDOW_DAYS = 30`、`WATCH_SCAN = 1000`：**只看最近 30 天内、最多 1000 条**。
- 后果：**更早的匹配不会被列出**；且若 30 天内新增超过 1000 条，只看最新的 1000 条。
- 依据：关注要的就是「新条目」，且这是**唯一**能把读取量与库大小解耦的办法（见 §4）。
- **页面明写了这两个数字**（模板直接插值 `${WATCH_WINDOW_DAYS}`/`${WATCH_SCAN}`，**不会与代码漂移**）。

## 6. 其它边界
- `cn_watch_seen` 曾**只增不减**（复核指出）。**实际发货的剪枝**（ack 时执行）是按**条目自身的窗口归属**判定，**不是**按 ack 时间：
  `DELETE FROM cn_watch_seen WHERE watch_id=?1 AND NOT EXISTS (SELECT 1 FROM cn_items i WHERE i.id=cn_watch_seen.item_id AND COALESCE(i.published_at,i.fetched_at) >= ?2)`
  （曾经写成 `... AND at < 窗口起点`，但那在 `published_at` 为**未来日期**时有洞——见 §3c#1。相关子查询按主键查 `cn_items`，**不用 join**，避开 D1 无 ANALYZE 时的 join 顺序陷阱。）
  它只会删掉「其条目已不在窗口内（或已从 `cn_items` 消失）」的已读行——这类条目**在重新进入窗口之前**不会再成为候选，其已读行是死重量。
  （**刻意不写「永不再成为候选」**：那是个**写下来即为假**的绝对断言，下一条就是反例。本轮的病根正是「假前提被写下并被信任」。）
- **一个良性的重复通知路径**（如实记录）：某 feed 条目若消失 30+ 天后又重新出现，`ON CONFLICT(id) DO UPDATE SET fetched_at`（worker_cloud.js:239）会抬高其 `fetched_at`；对 `published_at` 为空的条目这就是其 recency → 它会重新进入窗口并**再次通知**。这是**可接受的**：对关注而言它**确实又是新的**。
- 关注上限 `WATCH_MAX = 20`；`/api/watch/add` 对 facet 做**白名单**、value 长度截断 80 并**全部走 bind**。
- 新表由 `CREATE TABLE IF NOT EXISTS` **懒建**（沿用 T081 建 `cn_rum` 的既有模式），不需要单独的 D1 迁移；`runDaily` 不调用它，故建表失败不会影响每日 cron。
- `keyword` 现在是**共享扫描后在 JS 内 `includes`**（不再走 SQL LIKE）→ `%`/`_` 变成**字面量**（比之前更符合直觉），大小写不敏感。
