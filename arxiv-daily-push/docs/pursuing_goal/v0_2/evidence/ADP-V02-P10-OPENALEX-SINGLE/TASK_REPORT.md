## 病灶（实测，复核独立复现）## 病灶（实测，复核独立复现）
P08 用一个批量 `?filter=doi:a|b|c`（50 个 DOI，1 个子请求）。从 Cloudflare 边缘：

| | 边缘 | 本机 curl |
|---|---|---|
| `/works?filter=doi:…`（P08 的设计） | **429 ×3/3**，`retry-after≈46884s`，`Insufficient budget… $0 remaining. Resets at midnight UTC` | 200 |
| `/works/doi:X`（单条） | **200 ×12/12** | 200 |

OpenAlex 按 IP 计预算；Workers 出口是**共享数据中心 IP**，预算早被耗尽；`mailto` 不解决。
把发货的 `enrichMeta` 原样跑在真 D1 + 真 OpenAlex 上：`degraded:["meta:http429"]`、`total_rows: 0`。
**这不是偶发，是每晚都会发生的静默空转。**

## 修法（`test-results/proposed_fix_patch.mjs`，边缘实测可行）
批量 → **N 条并行的单条 `/works/doi:X`**；`META_BATCH(50)` → `META_PER_RUN(12)`（**1 DOI = 1 子请求**，
cron 20/50 → **32/50**）；404=确知未收录→写 `found=0`；429/5xx/异常=**不知道**→**什么都不写**；
`select=` 让响应显著变小；单条端点回**规范记录**，故 P08 的 F3「重复 work 最后一条赢」**不复存在**。
★关于「53671 → 1059」这个数★：它**是真的**，但**只对那一个 DOI**（`10.1056/nejmoa2512275`）——
我把**单个样本**当成了普适结论写出来，复核抽了别的 DOI 得到 17441–28437，于是显得对不上。
逐个复测（本轮）：

| DOI | 不带 select | 带 select | 倍数 |
|---|---|---|---|
| `10.1056/nejmoa2512275` | 53671 | 1099 | 48.8× |
| `10.64898/2026.03.04.709430` | 28351 | 981 | 28.9× |
| `10.48550/arxiv.1506.01497` | 17475 | 987 | 17.7× |

**诚实表述**：`select=` 把响应压到约 **1KB**，省下 **约 18×–49×**（随 DOI 的作者/机构规模而变）。
这与 P08 那张「重复率三个数字属于不同总体」的表是**同一类错**：**数是真的，总体没写清楚**。

**复核独立验证了前提，并把修好的版本跑在边缘上**：`requested:12, matched:12, degraded:[]`、1835ms
—— **正是 P08 失败的那个场景**。DIR-007 也被复核逐条数过：20 + 12 = 32/50 成立。

## 复核对「字段清单要不要在 CI 里重抓」的裁定（记为复核的理由，不是我的）
我把 `openalex_valid_select_fields.json` **钉住**（2026-07-17 抓取，含出处），没有放进 CI 每次重抓。

把活 API 的断言放进部署门，等于**拿一个已被覆盖的风险换一个不稳定的门**。
若要收掉这条残余，**放在非阻塞处**（例如一份定时的漂移报告），**绝不要放进门里**。
