# PFI v0.2.3 历史约束废弃政策

## 核心原则

历史资料只能作为证据和参考，不能自动继承。后续 Codex run 必须先判断历史内容属于以下哪一类：

- 已作废约束。
- 可保留工程纪律。
- 可参考 UX 方向。
- 必须排除的失败模式。

如果历史文档和 v0.2.3 Task Pack 冲突，以 v0.2.3 Task Pack、Roadmap 和本目录合同为准。

## 已作废约束

| 历史约束 | v0.2.3 结论 |
|---|---|
| 一级入口 9 个 | 作废。v0.2.3 固定 10 个一级入口。 |
| `市场与研究` 不得作为一级入口 | 作废。`市场与研究` 是正式一级入口。 |
| 暗色 AI 控制台风格 | 作废。默认明亮、高质感浅色系。 |
| 演示数据可用于财务验收 | 作废。禁止任何虚构财务数据进入验收。 |
| README/docs 写完成即可 closeout | 禁止。必须有 Evidence Pack 和真实验证。 |
| 旧 UI shell 继续补丁式堆叠 | 作废。后续 Stage 必须按人类产品体验恢复。 |

## 保留纪律

以下历史原则继续有效：

- 每轮只执行一个 Stage。
- 未经过用户验收不得进入下一 Stage。
- 禁止文档声明冒充完成。
- 禁止字符串/marker 测试冒充真实交互验收。
- 禁止 toast、localStorage 或浏览器缓存冒充真实保存。
- 禁止写死图表、写死趋势、写死净资产、写死投资市值。
- 禁止伪造截图、终端输出、路径、hash 或运行结果。
- 每个 Stage 必须生成 evidence pack。

## 冲突处理

发现历史约束冲突时，后续 run 必须：

1. 停止继承冲突约束。
2. 在 evidence 中记录冲突来源和处理结果。
3. 不以历史完成声明替代当前真实验证。

## Stage 11.2 Freeze

Stage 11 Phase 11.2 将本政策冻结为后续 v0.2.3 开发的硬约束：

- 冻结后不得恢复历史 9 入口约束。
- 冻结后不得把 `市场与研究` 从正式一级入口移除。
- 冻结后不得把历史完成声明写成当前用户验收。
- 冻结后不得用 README/docs 声明替代真实 evidence、测试输出、截图或 app/localhost 验证。
- 冻结后不得把虚构财务数据写入正式 UI、报告、首页摘要、图表、建议或验收证据。
- 冻结后每次 run work 最多只解决一个 phase；Stage 完成后才允许 whole-stage review。
