# 公开研究驱动的总系统升级方案

生成日期：2026-06-05

适用范围：PFIOS、AI-Research-System、消费分析系统、独立验证系统、政府政策系统、后续新增子系统。

## 1. 核心原则

所有系统升级必须基于公开可访问、用户授权或用户提供的信息。

允许使用：

- 官方文档、官方 API、GitHub、开源项目、学术论文、技术博客、竞品官网、公开数据集。
- 小红书、B站、知乎、公众号、WallStreet、moomoo 等公开或用户授权内容。
- 用户上传的 CSV、PDF、截图、Markdown、日志、代码和数据库。

禁止使用：

- 未授权账号数据。
- 未授权 cookie、token、密码。
- 绕过平台限制的抓取。
- 无来源、无验证、无法复现的结论。
- 为凑数量生成低价值策略、交易、文件或建议。

无法访问的信息统一标记为：

```text
无法访问 / 需要授权 / 需要用户手动导出
```

## 2. 研究输出标准

每个研究结论必须区分：

| 类型 | 含义 | 可用于决策程度 |
| --- | --- | --- |
| FACT | 来源直接支持的事实 | 高 |
| INFERENCE | 基于事实推导的判断 | 中 |
| OPINION | 观点或经验建议 | 低 |
| OBSERVATION | 当前本地系统观察 | 取决于验证 |

每条行动建议必须包含：

- 预期价值。
- 执行难度。
- 风险。
- 所需输入。
- 验证方式。
- 失效条件。

默认只输出少量高价值行动，不输出大量低价值待办。

## 4. 最高 ROI 升级清单

### P0：公开研究证据门

目标：

把所有新功能、策略、研究报告和系统建议都纳入证据门。

实施：

- 新增 `public_source_log` 数据结构。
- 每个结论必须记录来源、访问时间、证据层级、数据质量、授权状态。
- ResearchBus 新增 `evidence_layer`、`source_access_status`、`verification_method`、`failure_condition`。

预期价值：

- 降低无来源结论和错误决策。
- 让行研、政策、回测、消费画像的结论可追溯。

执行难度：

- 中。

风险：

- 早期录入成本增加。

所需输入：

- 公开 URL、用户上传文件、授权数据源。

验证方式：

- 任一报告必须能导出 source log。
- 缺来源时结论自动降级为 Observe 或 NeedsMoreEvidence。

失效条件：

- 来源无法访问且无本地缓存。
- 来源没有时间戳或不能证明数据口径。

### P0：Data Provider Registry 和数据质量评分

目标：

统一管理 moomoo、TuShare、AKShare、Yahoo Finance、Alpha Vantage、Polygon、OpenBB 等数据源。

实施：

- 为每个 provider 记录：市场覆盖、频率、是否授权、是否实时、延迟、复权口径、限制、最近成功时间。
- 回测前显示数据质量评分。
- 同一标的多源交叉校验结果写入报告。

预期价值：

- 解决“数据真实、及时、可靠、可验证、可追溯”的核心问题。

执行难度：

- 中。

风险：

- 免费数据源会受限、延迟或字段缺失。

所需输入：

- moomoo OpenD 状态。
- TuShare token。
- Alpha Vantage / Polygon key，如用户后续提供。

验证方式：

- 同一 symbol 至少输出主源、备源、差异、是否通过。

失效条件：

- 主数据源不可用。
- 多源价格差异超过阈值且无法解释。

### P0：策略实现风险交叉校验

目标：

同一策略逻辑在 PFIOS 内部引擎、向量化引擎、事件引擎之间做结果差异检查。

实施：

- 先选择 3 个标准策略：均线交叉、追跌杀涨、轮动策略。
- 输出收益、回撤、交易次数、成本、持仓路径差异。
- 记录差异来自：撮合口径、手续费口径、滑点口径、复权口径、买卖时间点。

预期价值：

- 避免因为回测引擎细节造成错误结论。

执行难度：

- 中高。

风险：

- 不同引擎模型天然不同，不能追求完全一致。

所需输入：

- 标准测试数据集。
- 明确成本和成交假设。

验证方式：

- 同一策略在零成本、同样价格口径下结果应高度接近。
- 非零成本下必须解释差异来源。

失效条件：

- 无法统一成交时间或复权口径。

### P1：分片验证升级为可选 worker 架构

目标：

保持子系统可独立运行，同时让 PFIOS 能调度百万、千万、亿级验证任务。

实施：

- 保留当前分片验证框架。
- 新增 worker manifest、shard manifest、result manifest、heartbeat。
- 本机默认使用轻量进程池。
- 数据量超过本机阈值时，提示需要 Ray/Dask 或外部机器，不伪装成可立即完成。

预期价值：

- 支持用户要求的大规模验证，同时避免本机资源被无意义耗尽。

执行难度：

- 中高。

风险：

- 大规模模拟如果没有缓存、采样和中止机制，会浪费时间和内存。

所需输入：

- 任务规模、分片大小、最大运行时间、是否允许外部 worker。

验证方式：

- 先通过 10 万、100 万、1000 万三档样例。
- 每档必须产出 manifest、失败分片、重跑命令。

失效条件：

- 本机内存不足。
- 用户不允许长时间后台任务。
- 数据源许可不允许批量抓取。

### P1：个人投资行为画像闭环

目标：

把消费分析、持仓、交易记录、策略回测、行研结论连接成个人行为改进系统。

实施：

- 消费系统输出现金流压力、冲动消费、固定成本、流动性压力。
- PFIOS 输出持仓集中度、回撤承受度、交易频率、补仓纪律。
- 行研系统输出研究证据充分度和反方观点质量。
- 总系统输出行为风险标签和优化建议。

预期价值：

- 直接提升真实投资决策质量，而不是只追求回测收益。

执行难度：

- 中。

风险：

- 行为建议可能被误读为交易指令。

所需输入：

- 用户上传持仓、交易流水、消费账单、研究记录。

验证方式：

- 每条行为建议必须能指向原始记录或统计指标。

失效条件：

- 持仓或消费数据过期。
- 用户未授权读取相关文件。

### P1：情绪分析升级为证据型情绪观察

目标：

让大盘、自选、持仓情绪分析更可靠、更可解释。

实施：

- 对公开新闻、公告、社区内容、价格趋势、波动率、RSI、VIX、资金流进行分层展示。
- FinBERT / FinGPT 只作为文本情绪模型候选。
- 小红书、B站、知乎、公众号等内容默认需要用户手动导出或授权。
- 输出 `情绪偏热 / 中性 / 偏冷`，不输出买卖指令。

预期价值：

- 让盘感训练和情绪观察有来源、有证据、有边界。

执行难度：

- 中。

风险：

- 社区情绪噪声大、易过拟合短期。

所需输入：

- 公开链接、手动导出的帖子/评论/视频文案、授权 API。

验证方式：

- 情绪分数必须展示样本数、来源分布、更新时间。

失效条件：

- 样本不足。
- 来源集中在单一平台或单一观点。

### P2：总系统研究雷达

目标：

定期扫描高价值公开开源项目、官方文档和竞品变化，把有价值机制变成候选需求。

实施：

- 每周生成 `ResearchRadar` 报告。
- 来源限制在官方/GitHub/论文/授权内容。
- 每个候选功能必须经过 ROI 评分。

预期价值：

- 在别人已有成果上持续升级系统，不重复造轮子。

执行难度：

- 中。

风险：

- 如果缺少筛选机制，会变成低价值信息堆叠。

所需输入：

- 关注项目列表、关键词、允许访问平台。

验证方式：

- 每周只允许进入 3-5 个候选项。
- 没有明确 ROI 的候选项不进入开发队列。

失效条件：

- 来源不可访问。
- 候选功能无法落地到本地系统。

## 5. 子系统新增能力建议

| 系统 | 新增能力 | 优先级 | 说明 |
| --- | --- | --- | --- |
| PFIOS | 公开研究证据门、数据源评分、策略实现风险校验、总系统研究雷达 | P0/P1 | 作为母系统统一治理 |
| AI-Research-System | 报告结论证据分级、反方观点评分、PFIOS 验证状态强引用 | P0 | 报告不能只写观点，必须带验证状态 |
| 消费分析系统 | 投资行为现金流压力、冲动风险、可投资现金流稳定性输出 | P1 | 给个人画像和仓位纪律提供依据 |
| 独立验证系统 | worker manifest、分片重跑、资源上限、结果摘要回写 | P1 | 保持独立运行，不写 PFIOS 主库 |
| 政府政策系统 | 政策来源权威评分、政策影响行业映射、政策证据可用性状态 | P1 | 缺授权时 fail-closed |
| 新增 Research Radar 子系统 | 公开项目/竞品/论文追踪、ROI 评分、开发候选队列 | P2 | 建议先作为 PFIOS 模块，不单独新系统 |

## 6. 不建议立即做的事项

| 事项 | 原因 |
| --- | --- |
| 直接迁移到 LEAN 或 Qlib | 成本高，容易破坏现有个人工作台；先吸收机制 |
| 直接引入完整 Dagster | 本地个人系统目前会增加维护成本；先用 manifest/SQLite |
| 直接做全自动社区抓取 | 授权、合规和反爬风险高；先支持手动导出和公开链接 |
| 直接把 AI 预测涨跌作为核心 | 金融信号噪声高，容易误导；先做证据门和风险门 |
| 直接承诺亿级任务本机运行 | 资源不确定；必须分片、采样、manifest 和可中止 |

## 7. 进入开发队列的门槛

任何新功能进入开发队列前必须满足：

1. 有公开、授权或用户提供的信息来源。
2. 有明确 ROI。
3. 有所需输入。
4. 有验证方式。
5. 有失效条件。
6. 不接实盘交易、不真实下单。
7. 不依赖未授权抓取。
8. 能通过 ResearchBus 或本地 manifest 留痕。

## 8. 推荐下一轮实现顺序

1. 在 ResearchBus 增加 `public_source_log` 和证据门字段。
2. 在 PFIOS 新增 `公开研究 / Research Radar` 状态页或数据文件。
3. 把本文件的行动建议转为跨系统 task queue。
4. 先实现 Data Provider Registry 和数据质量评分。
5. 再实现策略实现风险交叉校验。
6. 最后升级情绪分析和个人行为画像。
