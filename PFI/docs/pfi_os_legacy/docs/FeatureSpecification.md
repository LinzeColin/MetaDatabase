## 2. 单标的回测

入口：`单标的回测`

用途：在指定市场、标的、周期和日期区间内，运行单策略回测或双策略对比。

主要输入：数据源、市场、标的、周期、开始日期、结束日期、策略、策略参数、初始资金、佣金率、滑点、冲击成本。

主要输出：目标走势、策略收益率、相对收益、核心指标、策略诊断、权益与摩擦图、Bootstrap 鲁棒性、交易表、持仓表、信号表、Word 回测报告、数据质量报告。

关键指标：总收益率、年化收益率、Sharpe、最大回撤、胜率、交易次数、买入次数、卖出次数、买入持有收益、相对收益。

增删修补入口：可以新增策略、修改成本模型、调整指标展示、增加图表、优化报告模板。

风险点：真实数据源可能有缺口；分钟级数据受权限和噪声影响更大；策略表现不能直接视为未来收益。

测试方式：运行 `PYTHONPATH=src .venv/bin/pytest tests/test_backtest.py tests/test_app_dashboard.py -q`。

## 3. 情绪分析

入口：`情绪分析`

用途：观察大盘默认对象、自选对象或持仓对象的短期情绪温度，用于辅助判断研究环境。

主要输入：数据源、市场、对象来源、开始日期、结束日期、自选代码。对象来源可同时选择大盘对象、我的持仓和自选代码。

主要输出：已选择对象、情绪总览、情绪证据闸门、情绪结构分布图、情绪观察卡片、情绪分对比图、指标明细表、失败对象表。

当前指标：1 日涨跌、20 日涨跌、RSI14、相对 20 日均线、20 日年化波动率、60 日最大回撤、情绪分、情绪状态。

情绪分口径：基础分 50，叠加短期收益、20 日趋势、相对均线位置和 RSI，扣除高波动和较大回撤影响。波动率对象会反向处理。

状态解释：`极度低迷`、`偏冷`、`中性`、`偏热`、`过热`。

证据闸门：页面在展示具体对象前检查数据源、对象覆盖、失败率、样本长度、数据新鲜度和情绪集中度。出现 `Review` 或 `Block` 时，情绪分析只能作为观察线索，不能进入交易前参考。

增删修补入口：可加入新闻情绪、公告情绪、资金流、估值分位、行业热度、市场宽度、涨跌家数、成交拥挤度、持仓名称到基金/ETF 代码的映射维护。

持仓情绪联动：如果正式持仓缺少代码，系统会用本地代理规则把明显的基金名称映射到 ETF、指数或行业代理。代理只用于情绪观察，不代表基金本身。用户自定义映射文件为 `data/holdings/HoldingSymbolMap.json`。

跨系统映射：持仓名称到行情代理代码的结果会同步到 ResearchBus 的 `holding_symbol_mappings` 表，并由行研系统导出到 `data/report_artifacts/research_bus_bridge/HoldingSymbolMappingsFromBus.json`，方便日报、周报、行研验证和情绪分析使用同一套口径。

实体注册：持仓同步时会派生 `PFIOSEntityRegistryV1` 摘要，区分 `TradableSymbol`、`ProxyMapped` 和 `MissingSymbol`。`ProxyMapped` 代表研究代理，不等于真实可交易证券；`MissingSymbol` 不能进入回测、情绪或热点分析。实体注册可导出为 `data/entityRegistry/EntityRegistry.json`、`EntityRegistry.csv` 和 `EntityRegistry.md`，这些文件是派生产物，不覆盖正式持仓簿。

Workflow Layer：`workflow_inputs_frame()` 将自然语言聊天、dropbox 文件和直接 API 请求统一成只读可追溯输入视图。聊天输入状态会随 linked request 处理结果更新；持仓/交易候选进入 `PendingReview` 后必须确认，不能自动覆盖正式持仓；畸形 payload 会被拒绝入队，重复 dropbox 文件不会覆盖旧 processed 文件。

Report Evidence Layer：回测 Word 报告和 RunMetadata JSON 写入 `PFIOSReportEvidenceV1` 摘要，包含数据质量、多源交叉校验、实体状态、工作流输入来源、成本假设、风险闸门、决策质量和缺失证据。缺关键证据时，报告证据状态降级为 `NeedsMoreEvidence`。

最终集成审计：`scripts/auditPFIIntegration.sh --no-write` 只读检查 Data Trust、Entity Registry、Workflow Inputs、Report Evidence、ResearchBus 互通和禁止实盘边界，输出 `Pass`、`Review` 或 `Fail`。该审计用于日常启动前快速判断系统证据链是否闭合。

风险点：情绪分是技术状态摘要，不是涨跌预测，也不是交易指令。

窗口口径：`展示开始日期` 只控制页面查看范围。系统会自动向前扩展指标预热窗口，用更长历史计算 RSI、波动、回撤和情绪分，避免同一目标日期因为展示开始日期不同而发生不合理跳变。

测试方式：运行 `PYTHONPATH=src .venv/bin/pytest tests/test_sentiment.py -q`。

## 4. 热点分析

入口：`热点分析`

用途：用热力图、气泡图和时间切片观察大盘、行业、风格、避险资产和持仓对象的短期强弱扩散。

主要输入：数据源、市场、时间粒度、分析范围、开始日期、结束日期、自选代码、是否每小时自动刷新。

主要输出：分析对象表、热点时间轴、时间切片、热点总览、热点证据闸门、热力图、气泡图、优先复核对象表、热点明细表、失败对象表和热点分析口径。

当前指标：近 1 期涨跌、近 5 期涨跌、近 20 期涨跌、RSI14、20 期波动、20 期回撤、即时热度、平滑热度、热度变化、热点状态、气泡大小。

热点热度口径：基础分 50，叠加近 1 期、近 5 期和近 20 期涨跌、RSI，扣除过高波动和较大回撤。VIX 等波动率对象会反向处理，因为波动率上升通常代表风险温度上升。系统会先得到单个时间切片的 `即时热度`，再用最近数个切片生成默认展示的 `平滑热度`；平滑热度用于减少小时级噪声，热度变化用于观察持续扩散或降温。

刷新口径：热点数据读取函数使用 3600 秒缓存 TTL；开启自动刷新后，页面每小时刷新一次当前页，以触发缓存失效后的重新读取。真实数据是否更新取决于数据源权限、行情延迟和交易时段。热点时间轴基于已读取结果生成，不额外重复请求数据源。

快速预检：点击 `生成热点分析` 前页面先显示 `PFIOSHotspotQuickPreflightV1`，只基于已选择对象、工作台模式和缓存 metadata 判断 `Ready`、`CacheHit`、`LargeRun` 或 `NeedsInput`，展示预计 provider 请求数、切片上限、缓存命中状态和下一步建议。预检不读取行情、不生成热点、不连接券商、不创建订单、不修改持仓。

运行摘要：点击 `生成热点分析` 后页面生成 `PFIOSHotspotRuntimeSummaryV1` compact 摘要，包含请求指纹、对象覆盖、失败对象、切片数量、缓存 TTL、证据状态和只读安全边界。摘要只从已缓存的热点结果生成，不保留原始行情明细，不连接券商，不创建订单，不修改持仓。

持久缓存：热点页面使用 `PFIOSHotspotPersistedCacheV1` 把已计算的热点结果写入 `data/cache/hotspots/`，同一请求指纹在 TTL 内可直接复用，减少工作台重启或 Streamlit 内存缓存失效后的重复行情请求与重复热度计算。该缓存是本地派生结果，被 Git 忽略，不存 secrets、原始账户数据、券商 token、订单或持仓修改。

52ETF 对照：`scripts/site52etfSnapshot.sh` 可生成 `PFIOS52ETFPublicSnapshotV1` 本地 latest snapshot，记录 52ETF 公开 A 股云图板块、指标、操作提示和只读证据闸门。勾选 `显示 52ETF 公开参考` 后，热点页面优先读取本地 latest，缺失时才按缓存在线读取公开页面，并生成 `PFIOS52ETFHotspotComparisonV1` 对照摘要，把 52ETF 公开 A 股云图板块映射到 PFI_OS 当前热点对象池。该对照只用于 UI/对象池复核，不写入回测、不触发交易、不替代本地数据质量闸门。

图表交互口径：热点时间轴、热力图、气泡图和向量化研究图表使用统一 Plotly 研究图表配置，默认开启滚轮缩放、拖拽平移或缩放、悬停辅助线、响应式布局和 PNG 导出。时间轴额外提供范围滑块和快捷区间按钮。图表操作只改变前端视图，不触发行情请求、参数扫描、回测、券商连接、下单或持仓修改。

证据闸门口径：热点页面会检查数据源、数据覆盖率、失败率、样本长度、时间切片、刷新粒度、数据新鲜度和热度集中度。任一检查项为 Review 或 Block 时，页面结论只能作为研究观察，不能作为交易前参考。

窗口口径：热点页面把 `指标预热窗口` 和 `展示窗口` 分离。真实请求会从展示开始日期之前自动扩展，热力图、气泡图和时间轴只展示用户选择区间内的时间切片。同一对象、同一数据源、同一时间切片在不同展示窗口下应保持同一即时热度、平滑热度和热度变化。

横截面口径：热点比较必须在同一个横截面时间下进行。不同数据源或不同市场对象可能返回不完全一致的小时戳，系统会以页面选择的时间切片为统一观察点，对每个对象取该时间点之前最新可用行情参与比较，并在明细表保留 `实际行情时间`，避免因为时间戳不齐导致平均热度或对象数量异常跳变。

增删修补入口：可加入真实资金流、行业涨跌家数、市场宽度、成交额排名、新闻催化、估值分位、热点持续性评分和报告嵌入。

风险点：热点热度是横向技术状态，不是涨跌预测；单小时异动可能来自新闻、资金流或数据异常，必须结合数据质量、基本面和外部事件复核。

测试方式：运行 `PYTHONPATH=src .venv/bin/pytest tests/test_market_hotspots.py tests/test_app_dashboard.py -q`。

## 5. 盘感训练

入口：`盘感训练`

用途：用技术面指标训练读图能力，把一个对象的价格状态拆成趋势、动能、风险、支撑压力和量价确认模块，并通过隐藏答案区间训练方向、收益区间和精确涨跌幅判断。

主要输入：数据源、市场、对象来源、开始日期、结束日期、训练场景、训练难度、判断周期、限时秒数、自选代码。对象来源可同时选择大盘对象、我的持仓和自选代码。

主要输出：盘感训练卡片、隐藏答案区间的技术图表、实时自动倒计时、限时判断题、答案揭示、完整走势图、指标讲解表、事前技术分析、多维复盘、训练明细表、失败对象表。

当前指标：1 日涨跌、5 日涨跌、20 日涨跌、60 日涨跌、MA20、MA60、20 日支撑、20 日压力、RSI14、MACD、Bollinger、ATR14/价格、20 日年化波动、成交量比、60 日最大回撤。

盘感分口径：基础分 50，叠加价格相对 MA20/MA60、20 日和 60 日趋势、RSI、MACD、成交量确认，扣除 Bollinger 极端位置、ATR、波动率和回撤风险。分数用于训练读图，不代表涨跌预测。

状态解释：趋势判断包含 `多头结构`、`短期改善`、`震荡结构`、`短期承压`、`空头结构`；动能判断包含 `动能拥挤`、`动能偏强`、`动能中性`、`动能偏弱`、`动能低迷`；风险判断包含 `风险平稳`、`风险偏高`、`风险升温`。分析文本会把 MA、支撑压力、RSI/Bollinger、MACD、ATR/波动/回撤和成交量转换成自然语言技术判断，帮助复核结论来源。

训练难度：`入门` 只判断未来指定周期的方向；`中等` 判断方向和收益区间；`专家` 判断方向、收益区间和更精确涨跌幅。点击开始后页面会实时倒计时；系统记录用时、是否超时、方向是否正确、区间是否正确和专家数值误差。

复盘口径：事前技术分析只使用答案揭示前的数据。如果技术面倾向和真实走势不一致，系统必须提示补充事实面、基本面、价值面、交易面和市场环境证据，不能根据结果倒推当时不存在的分析理由。

增删修补入口：可加入 K 线形态识别、支撑阻力、市场宽度、行业相对强弱、盘感训练记录、错题复盘和报告嵌入。

风险点：盘感训练是技术状态解释，不是交易建议；强势可能延续也可能拥挤，低迷可能反弹也可能继续恶化；真实研究仍需数据质量、多源校验、基本面和风险门禁支持。

测试方式：运行 `PYTHONPATH=src .venv/bin/pytest tests/test_market_feel.py tests/test_app_dashboard.py -q`。

## 6. 持仓

入口：`持仓`

用途：读取、同步、保存和维护个人持仓，区分正式持仓、截图候选持仓和待确认订单。

主要来源：支付宝持仓账本、行研报告上传目录、消费行为分析系统目录、PFIOS 本地导入目录、手动录入。

正式持仓簿：`$PFI_OS_HOME/data/holdings/HoldingsBook.json`

同步历史：`$PFI_OS_HOME/data/holdings/HoldingsImportHistory.json`

导入目录：`$PFI_OS_HOME/data/holdings/imports`

主要字段：代码、名称、市场、数量、成本、持仓金额、浮动盈亏、权重、更新时间、来源系统、来源文件。

当前规则：待确认订单不进入正式持仓；截图候选持仓不进入正式持仓；存在有效市值时系统按市值重新计算权重。

主要输出：当前持仓表、暴露拆解、同步来源、同步历史、待确认订单、手动维护表单、质量检查。

当前持仓表展示规则：隐藏来源文件 path；显示来源系统、代码、名称、市场、数量、成本/成本价、持仓金额、持有收益、持有收益率、权重和更新时间。

持有收益率口径：`持有收益率 = 持有收益 / (持仓金额 - 持有收益)`。分母不可用时显示 `0.00%`。

增删修补入口：可接入券商导出 CSV、基金确认份额、账户总资产、现金余额、汇率、基金穿透、主题暴露和行业暴露。

风险点：如果来源文件为空，系统不会假造持仓；不同系统的基金名称和代码可能不一致，需要人工复核。

测试方式：运行 `PYTHONPATH=src .venv/bin/pytest tests/test_holdings_book.py -q`。

## 7. 报告中心

入口：`报告中心`

用途：管理 PFIOS 生成的 Word 报告、运行元数据、数据质量文件、实验记录、复盘记录和验证任务。

主要输入：报告目录、日期筛选、关键词、报告类型、运行状态、研究状态。

主要输出：报告资产总览、最新报告、运行判读、决策质量 Dashboard、验证任务队列、复盘错误画像。

核心文件目录：`~/Downloads/量化回测分析/YYYY-MM-DD/`

增删修补入口：可增加全文搜索、报告评分、报告标签、证据链图谱、批量导出、报告归档。

风险点：报告结论必须结合数据质量、成本假设和风险门禁，不能只看收益率。

测试方式：运行 `PYTHONPATH=src .venv/bin/pytest tests/test_reports.py tests/test_report_catalog.py -q`。

## 8. 行研报告

入口：`行研报告`

用途：按日期和关键词索引本地行研报告，并通过统一研究数据总线联动 PFIOS 报告、验证任务、研究门禁状态、持仓主数据、独立验证运行状态、FIFA/TAB 研究系统和政府文件/政策解读系统产物。

默认目录：`~/Downloads/行研报告`

主要输入：开始日期、结束日期、关键词、报告文件。

主要输出：报告列表、报告路径、最近 PFIOS 报告、验证任务队列、最近研究门禁状态、共享 SQLite/JSON 快照、子系统注册表、子系统产物索引、母系统编排记录。

持仓联动输出：行研系统可读取 `HoldingsMasterFromBus.json`、`HoldingSymbolMappingsFromBus.json` 和 `PortfolioTransactionsFromBus.json`，用于把持仓、交易候选、情绪代理和回测验证连接到同一条证据链。

互通审计：`scripts/auditResearchBusInterop.sh --json` 输出 `data/researchBus/ResearchBusInteropAudit.json`。当前审计范围覆盖共享 schema、共享 SQLite、系统注册表、子系统产物索引、双向 API、任意聊天输入、行研任务解析、PFIOS 结果回写、消费行为同步、持仓主表、独立验证两级架构、系统心跳、行研桥接和行研 automation 保留状态。

母子系统编排：`scripts/orchestrateSystems.sh` 支持 `register`、`sync-artifacts`、`status` 和 `run`。默认 dry-run 只登记将调用的命令；显式加 `--execute` 才实际运行子系统。FIFA 和政策系统保留自身独立入口，PFIOS 只作为母系统登记、调度和同步产物。

增删修补入口：PDF/文本正文解析和验证任务自动提取已具备基础能力；后续可增加 OCR、语义去重、报告摘要质量评分、证据链图谱。

风险点：PFIOS venv 当前缺少 PDF 解析库时只能索引 PDF；行研系统侧已通过 `pypdf` 承担 PDF 正文解析。扫描图片 PDF 仍需要 OCR。

测试方式：运行 `PYTHONPATH=src .venv/bin/python -m pytest tests/test_system_orchestrator.py tests/test_independent_validation.py tests/test_research_bus_audit.py tests/test_research_bus_api.py tests/test_research_bus.py -q -p no:cacheprovider`，行研系统运行 `python3 -m unittest tests.test_research_bus_bridge -v`，真实状态运行 `scripts/auditResearchBusInterop.sh --json --no-write`。

## 10. 参数扫描

入口：`参数扫描`

用途：比较一组参数组合，判断策略是否只依赖单个偶然参数。

主要输入：标的、市场、日期、短均线参数、长均线参数、成本、初始资金。

主要输出：参数扫描预检、参数结果表、热力图、Top 参数组合、稳定性分析、Train-Test 验证、Walk-Forward 验证、实验报告。

预检口径：页面生成 `PFIOSParameterScanPreflightV1`，只解析参数网格和选择项，不读取行情、不运行回测。预检状态包括 `Ready`、`LargeRun`、`TooMany`、`InvalidGrid` 和 `Blocked`，用于提前提示组合数、最大组合数、上限占用、网格错误或标的错误。

增删修补入口：可扩展到 RSI、BOLL、MACD、KDJ、ATR、多指标组合和自定义策略参数。

风险点：参数扫描越多，越容易数据挖掘；必须配合样本外验证。

测试方式：运行 `PYTHONPATH=src .venv/bin/pytest tests/test_experiments.py tests/test_app_dashboard.py -q`。

## 16. 增删修补功能时的固定流程

1. 明确功能属于哪个功能区，或是否需要新增功能区。
2. 写清楚输入数据、输出结果、保存路径、计算公式和风险假设。
3. 修改代码前先确认会影响哪些文件和哪些测试。
4. 实现后运行相关测试。
5. 更新 README、QuickStart、Handbook、ReportGuide、Testing、AcceptanceChecklist 和本功能说明书。
6. 说明风险、回滚方式和下一步建议。
7. 如果是正式说明书快照，重新生成 `FeatureSpecification.pdf`。

## 17. 需求变更模板

功能区：填写单标的回测、情绪分析、持仓、报告中心等。

操作类型：新增 / 删除 / 修改 / 修补 / 优化布局 / 优化报告 / 接入数据源。

目标：一句话说明你希望这个功能解决什么问题。

输入：需要哪些数据、文件、账号、API Key 或手动填写字段。

输出：页面展示、报告、表格、图表、JSON、CSV、Word 或 PDF。

保存：是否需要永久保存，保存到哪个目录。

规则：涉及金额、成本、收益、仓位、税费或汇率时，必须写清公式和假设。

验收：你希望看到什么结果才算完成。

风险：哪些情况必须阻止运行、提示证据不足或降级为观察。
