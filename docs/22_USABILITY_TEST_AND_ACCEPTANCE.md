# 22 - 可用性测试与 UI/UX 验收

## 1. 核心任务

| 任务 | 成功标准 |
|---|---|
| 从打开产品到看到 NVIDIA 商业版图 | 无中间卡片页；首屏主图可见 |
| 找到 NVIDIA 的关键上游 | 30 秒内找到并解释至少两个上游连接 |
| 把 TSMC 设为研究中心 | 两次以内明确操作；无需双击/右键 |
| 连续切换三个主体 | breadcrumb、返回和状态继承正确 |
| 只看供应链 | 一次切换后非供应链关系明显降噪 |
| 查看关系依据 | 从节点/关系到来源不超过两步 |
| 保存并恢复视图 | 刷新或重开后主体、视角和路径恢复 |
| 不看图完成核验 | 可用列表模式查到同一关系与证据状态 |

## 2. 定量门槛

- 首屏中央可视化区域占应用可用面积至少 60%。
- 有 Watchlist 时，打开至主图可交互的 P75 小于 2.5 秒（本地 fixture）。
- 首次用户 3 秒内能指出当前主体；30 秒内能指出上游和下游方向。
- “以它为中心”任务成功率至少 90%，中位操作数不超过 2。
- 三次连续切换后状态恢复测试 100% 通过。
- 100 节点 / 300 边局部图中，拖动、缩放、选择 P95 帧时间符合项目性能门槛。
- 键盘可以访问全部主动作；可见焦点不被遮挡。
- 指针目标通常不小于 24×24 CSS px；密集图节点提供等价列表操作。

## 3. 视觉回归

固定 1440×900、1280×800、1024×768 三档截图；验证左栏、画布、详情栏、时间条没有遮挡。对以下状态建立基准图：默认 NVIDIA、供应链视角、TSMC 切换后、详情栏关闭、列表模式、空 Watchlist、加载中、错误和无数据。

## 4. 内容验收

运行 UI 文案静态检查，主 UI 不得出现 `reroot`, `hop`, `profile`, `calibration`, `agent`, `gate`, `reported`, `derived`, `disputed`, `unknown`。开发者工具和日志可保留内部字段。

## 5. 研究测试

至少 5 名目标用户完成任务测试；记录任务成功、首击、错误、停顿、口述理解和主观信心。每个高严重度问题必须关联修复任务和回归测试，不以“用户需要培训”替代产品改进。
