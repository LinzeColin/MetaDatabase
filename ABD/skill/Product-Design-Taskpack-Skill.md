# Product-Design-Taskpack Skill｜完善版

## 目标

把一个尚有歧义的产品构想转成可由人类理解、由机器连续执行、由证据验收、可安全发布和长期运行的双平面任务包。最终人类平面严格7份文档，机器平面包含唯一事实、验收合同、任务图、持续证据和安全发布。

## 输入合同

至少接收：

- 战略目标和目标曲线；
- 用户、问题、价值和痛点；
- 资金、时间、基础设施和费用约束；
- 产品职责与明确非目标；
- 正常用户动作；
- 市场、数据、模型、来源和动作范围；
- 安全、隐私、监管和来源合同；
- 开发授权边界、暂停条件和终止条件；
- 最终版本号。

缺失信息先建立假设登记；可逆、普通、零成本问题采用明确默认，不阻塞开发。只有不可逆、法律/安全、超范围或超预算问题形成最小决策包。

## 执行顺序

### 1. Working Backwards

先写：

1. 面向客户的新闻稿；
2. 客户常见问题；
3. 可观察客户结果；
4. 失败时客户体验。

禁止先从技术架构开始。

### 2. 只读研究

先搜索官方文档、监管来源、研究论文、GitHub和公开软件系统。研究范围只围绕目标产品，不遍历用户无关仓库。

每条来源必须记录：

```text
来源编号
标题
URL
类型
检索日期
支持的主张
采用/适配/拒绝
许可证
版本或更新状态
不确定性
复查时间
```

不得声称“穷尽互联网”。必须记录查询、未覆盖区域和低置信结论，使调研可重复。

### 3. Canonical Facts

建立唯一机器事实并规定优先级。旧文档、旧包、用户新裁定和外部事实冲突时：

1. 最新明确用户裁定；
2. 当前版本机器事实；
3. 官方来源；
4. 已验证研究；
5. 历史文档。

人类文档由机器事实生成，禁止独立修改。

### 4. Product Requirements

每条需求必须有：

```text
唯一编号
问题
用户
价值
范围
非目标
现状基线
目标
测量
观察周期
唯一主验收
优先级
```

不允许一条需求多个含糊主验收，也不允许没有需求来源的开发任务。

### 5. Acceptance Contract

每个关键验收强制包含：

```text
环境
输入
可执行验收判定器
测试方法
阈值
证据
通过门
回滚
暂停条件
保留策略
```

阈值边界必须包含：

- 正好等于阈值；
- 略低；
- 略高；
- 缺失；
- 错误类型；
- 重复；
- 乱序；
- 时间边界；
- 故障；
- 恶意输入；
- 用户指定的敏感扰动。

### 6. 机器任务图

层级：

```text
Stage
  └─ Phase（每Stage默认≤5）
       └─ Task（每Phase默认≤5）
```

叶子Task建议0.5–4小时，包含：

```text
输入
输出
依赖
需求
验收
测试
判定器
环境
阈值
证据
风险
回滚
暂停条件
验证
通过门
工时区间
置信度
```

任务图必须：

- 无循环；
- 无孤儿；
- 所有依赖存在；
- 关键路径可计算；
- 普通任务通过后自动推进；
- 人类输入在开发前准备或建成非阻塞降级；
- 失败先回滚或降级，只有暂停条件才提问。

### 7. 双流水线

人工智能产品必须同时有：

**软件正确性流水线**

```text
Schema/类型
→ 单元/属性
→ 合同夹具
→ 集成/端到端
→ 全量/压力/并发/极限
→ 故障注入/恢复
→ 安全分析/组件/秘密/制品来源
→ 小流量试运行
→ 发布后测试
```

**模型能力与安全流水线**

```text
数据时间血缘
→ 市场/规则基线
→ 挑战模型
→ 时间外推
→ 校准与不确定性
→ 业务指标和成本
→ 敏感性/离群/容量
→ 红队/不同模型互审
→ 模型系统说明卡
→ Alpha/Beta/GA
→ 漂移/回滚
```

两条流水线独立通过；不得用软件上线替代模型有效，也不得用历史模型指标替代当前可执行性。

### 8. 路径穷举

每个业务流至少包含：

- Walking Skeleton；
- Golden Path；
- Black Path；
- Abuse Path；
- Degraded Path；
- Recovery Path。

每条路径要有输入、状态转换、输出、证据、用户动作和恢复。

### 9. Security Assurance

覆盖：

- 设计：威胁模型、信任边界、最小权限；
- 构建：安全编码、静态分析、依赖、秘密、软件物料清单、构建证明；
- 测试：越权、恶意文件、提示注入、数据投毒、身份错配；
- 维护：补丁时限、组件禁用、秘密轮换、事故与回滚。

外部网页、邮件、附件、评论和模型输出一律按不可信数据处理，不能修改机器真源、权限、资金事实或删除规则。

### 10. Continuous Evidence

每次任务、测试、模型、建议和发布保存：

- 输入哈希；
- 参数哈希；
- 代码哈希；
- 模型哈希；
- 输出哈希；
- 测试命令和退出码；
- 时间；
- 环境；
- 回滚结果。

固定追踪链：

```text
需求 → 验收 → 任务 → 测试 → 证据 → 制品 → 发布
```

### 11. Safe Release

强制：

- 功能开关；
- 双环境切换；
- 小流量试运行；
- 发布后探针；
- 自动回滚；
- 过期状态作废；
- 主动故障注入；
- 常态恢复演练。

### 12. 经济性与证伪

收益使用区间、敏感性、机会成本和置信度。禁止承诺伪精确收益。

必须区分：

- 目标；
- 可行性；
- 已验证结果；
- 建议表现；
- 真实资金表现；
- 非现金复用价值。

每个重要目标必须有：

- 可行性门；
- 证伪样本和观察周期；
- 终止条件；
- 不达标后的安全行为；
- 不允许为了达标改变的纪律。

## 双平面7文件

固定：

1. `00_我在哪.md`
2. `01_产品需求.md`
3. `02_系统架构.md`
4. `03_口径字典.md`
5. `04_操作流程.md`
6. `05_执行与验收.md`
7. `06_运维手册.md`

不得增加第8份人类治理文档。新增结构化信息进入机器平面，再渲染到上述7份。

## 中文和上下文效率

- 日常界面使用用户语言；
- 必要缩写第一次出现必须给中文名；
- 建议卡只保留做决定所需字段；
- 错误必须给唯一原因和下一步；
- 机器任务包含完整上下文，开发线程无需反复查找；
- 暂停时只输出一个最小决策问题、证据、默认建议和不决策后果。

## 最终交付门

只有全部通过才定版：

- 恰好7份人类文档；
- Canonical Facts无冲突；
- 每条需求恰好一个主验收；
- 每个验收有判定器、阈值和证据；
- 任务图无循环、无孤儿；
- 追踪矩阵闭合；
- 软件和模型双流水线完整；
- 安全保障覆盖全生命周期；
- 全量、压力、并发、极限、混沌和恢复已进入任务；
- 所有普通依赖已准备或有非阻塞降级；
- 版本、清单、哈希和校验报告一致；
- 最终只输出：一个任务包压缩包、一个Stage/Phase Roadmap、一句Pursuing Goal Prompt。
