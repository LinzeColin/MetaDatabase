# 18 - UI/UX 公开产品与规范调研

## 1. 结论

本项目不应继续采用“行业卡片目录 + 多段文字说明”的 Agent 式界面。默认产品形态应是一个**对象中心、Watchlist 驱动、关系画布优先的视觉调查工作台**：稳定左导航、中央关系画布、右侧按需详情、底部时间变化，并允许任何可探索对象被设为新的研究中心。

本轮是面向产品决策的广泛公开资料基准研究，不宣称穷尽整个互联网。研究覆盖：人机交互原则、无障碍标准、复杂应用壳层、专业画布工具、Watchlist 工作流、图谱探索产品、大图工程和 UI 内容设计。机器可读来源见 `data/uiux_benchmark_sources.csv`。

## 2. 研究问题

1. 用户打开产品时，怎样立即看到正在研究的公司，而不是先穿过目录？
2. 怎样用图形表达上下游、业务、资本、控制、政策和时间，而不是把它们写成卡片？
3. 怎样允许 NVIDIA -> TSMC -> ASML -> 材料供应商连续切换主体，同时不丢失上下文？
4. 怎样防止大型商业网络变成不可读的 hairball？
5. 怎样把证据、不确定性和数据滞后写成人类可理解的语言？
6. 怎样让键盘、触屏和不依赖颜色的用户也能完成同一研究任务？

## 3. 参考模式与采用决策

| 参考体系 | 观察到的成熟模式 | 本项目采用 | 明确不照搬 |
|---|---|---|---|
| Nielsen Norman | 状态可见、真实世界语言、用户控制与撤销、识别优先 | 明确当前主体、截至日期、breadcrumb、返回、保存与恢复；禁用内部术语 | 不把工程流程暴露给最终用户 |
| W3C WCAG 2.2 | 焦点、目标尺寸、非文本对比、非文本替代 | 键盘可达、可见焦点、24px 基线、线型/标签/图例、等价列表 | 不以“图很复杂”为由豁免替代操作 |
| Carbon UI Shell | 持久顶部与左侧导航适合复杂产品的频繁切换 | 稳定左导航 + Watchlist 上下文 + 顶部全局搜索 | 不建立三层以上导航树 |
| Figma 类画布工具 | 左侧上下文、中央画布、右侧所选对象属性 | 保持稳定空间骨架；选中对象才出现详情 | 不复制设计工具的高密度专业控件 |
| TradingView 类研究入口 | Watchlist 支持持续回访；对象切换更新主分析区 | 有 Watchlist 时直接打开最近或变化最重要的对象 | 不复制交易终端的报价密度和交易暗示 |
| Neo4j Bloom | 选择、检查、按类型/方向展开局部邻居 | 单击选择；右侧检查；“以它为中心”；上游/下游增量展开 | 不暴露标签、查询语言和图数据库术语 |
| Kumu | focus、filter、clustering 在同一地图上控制复杂度 | 分析镜头、局部聚焦、聚合组、保存当前场景 | 不让自由力导向布局承担全部语义 |
| Cytoscape.js | 大而密的图无论布局都容易变成 hairball，应展示相关子图 | 每个主体限制首屏关系预算；分组、排名和懒加载 | 不承诺一次显示“全部商业帝国” |
| Sigma.js | reducers 可在不修改底层数据时改变视觉强调 | 镜头切换只过滤/弱化关系，保持主体和上下文 | 不把视觉弱化误写为数据删除 |
| Palantir Object Explorer | 对象搜索、可配置可视化、表格、对象详情、保存探索 | 统一对象详情与保存场景；图和表共享状态 | 不复制后台配置复杂度 |
| Microsoft 内容规范 | 控件文案简洁、具体、以动作动词表达并保持术语一致 | “以它为中心”“展开上游”“保存视图”“查看来源” | 禁止“执行任务”“运行 Agent”“提交流程” |
| Shneiderman | Overview first, zoom and filter, details on demand | L0-L3 语义缩放和按需证据详情 | 不在首屏同时展示所有字段 |

## 4. 关键产品推论

### 4.1 首页就是研究现场

默认路由不是行业页。优先恢复上次场景；其次打开 Watchlist 中最近或未读变化最重要的公司；只有 Watchlist 为空时，行业才以画布内的轻量入口出现。

### 4.2 一张图不能承担所有分析任务

“商业帝国”是统一工作台，不是单一力导向图。系统使用多个视觉镜头：商业全景、供应链分阶段流图、集团层级、所有权/控制 DAG、资本流与并购、政策双向关系、战略变化和时间差异。它们共享同一主体、时点、筛选和探索路径。

### 4.3 递归探索必须是明确动作

单击节点只选择和查看详情，不立即重排。用户确认“以它为中心”后才切换主体。重绘时继承镜头、时间、筛选、分析偏好和 breadcrumb，并尽量保持仍相关节点的空间位置。

### 4.4 复杂度必须受预算约束

首屏只显示最关键的 20-40 条连接；同类对象超过 6 个优先聚合；每次展开默认返回 10 个高相关对象；其余内容通过继续展开、搜索或列表查看。全局 hairball 不是“全面”，而是信息失败。

### 4.5 文案属于信任模型

最终用户看到“公开披露 / 多源推断 / 信息冲突 / 尚未披露”，同时可展开查看内部关系类型、来源、报告期和计算方法。重要性评分必须说明依据，不得暗示收益概率或法律控制权。

## 5. 被否决的界面模式

- 独立行业卡片首页阻挡进入研究。
- 把 6-10 个指标卡当作主要信息表达。
- 一个无限扩张的全局关系图。
- 单击节点后跳到静态公司详情页，无法递归切换主体。
- 只用颜色区分关系或证据状态。
- 需要 hover、右键、双击或拖拽才能完成核心任务。
- 把 `reroot`、`hop`、`profile`、`calibration`、`agent`、`gate` 放进主 UI。

## 6. 验证方式

设计结论不是凭截图批准。必须通过 1440x900、1280x800、1024x768 视觉回归，以及“找到上游 -> 查看证据 -> TSMC 设为中心 -> 再切换两次 -> 返回 -> 保存并恢复 -> 列表核验”的任务测试。详细门槛见 `docs/22_USABILITY_TEST_AND_ACCEPTANCE.md`。
