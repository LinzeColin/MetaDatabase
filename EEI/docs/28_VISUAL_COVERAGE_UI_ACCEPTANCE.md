# 28 - 首页 90%+、全系统 80%+ 可视化覆盖与 UI 验收

## 1. 指标定义

“可视化 90%”不能只靠主观判断。采用两类指标共同验收。

### 1.1 视觉承载面积比

```text
VisualSurfaceRatio =
  union(area of graph/chart/map/timeline/matrix/visual KPI regions)
  / information-bearing workspace area
```

不计入：全局导航、纯工具栏、浏览器边框。文本标签位于图内时计入视觉区域；大段说明面板不计入。

- 默认首页目标：`>= 0.90`；
- 所有 P0 研究页面平均：`>= 0.80`；
- 模型中心因需编辑表单，目标：`>= 0.65`，但必须有公式图、权重图和影响预览；
- 操作日志可表格优先，不计入系统视觉页面分母。

### 1.2 功能视觉优先覆盖

```text
VisualFirstCoverage =
  P0 modules whose primary task is completable from a visual-first view
  / all P0 research and data-analysis modules
```

目标 `>= 0.80`。本方案 12 个 P0 分析模块中至少 10 个视觉优先。

## 2. 首页验收

- 首屏无需滚动即可看到当前主体和主关系图；
- 中央关系图 + 底部时间/数据条 + 右侧视觉摘要占信息区域 90% 以上；
- 大段公司介绍不超过 3 行；
- 关键指标以环、条、热力、迷你趋势或关系编码表达；
- 首页可直接完成选择、展开、切换主体、切镜头、调参数和保存；
- Watchlist 与数据库状态常驻但不抢占主图。

## 4. 自动化测量

Playwright 读取标记为 `[data-visual-surface]` 的区域，计算矩形并集和工作区面积；同时检查：

- 主图面积；
- 主体可见；
- 首屏文本块高度；
- 图形元素数量；
- 页面溢出和遮挡；
- 1440x900、1280x800、1024x768；
- reduced-motion；
- 125% 缩放。

## 5. 商务专业风格门槛

- 使用 8px 网格和统一密度；
- 圆角以 6-12px 为主，禁止大量胶囊和玩具化图标；
- 主色克制，状态色只用于状态；
- 阴影层级不超过 3；
- 页面文字层级不超过 5；
- 同一屏最多一个主行动；
- 图例和数据状态固定位置；
- 表格只承担核验，不抢占首页；
- 所有数字有单位、期间和来源语义。

## 6. 交互流畅验收

| 指标 | 门槛 |
|---|---:|
| hover/focus 反馈 | < 80ms |
| 点击选择反馈 | < 100ms |
| 详情抽屉开始显示 | < 120ms |
| 当前视图参数预览 p95 | < 250ms |
| 一层展开 p95 | < 700ms |
| 主体切换动画 | 320-420ms |
| 帧率 p95 | >= 55fps |
| 首屏 LCP（本地基准） | < 2.5s |
| 布局偏移 CLS | < 0.1 |

## 7. 失败条件

出现以下任一情况不得通过：

- 首页变成卡片仪表盘或文字档案页；
- 图谱只是背景装饰，关键任务必须去表格完成；
- 参数改变后只有一个图更新，其他视图仍使用旧版本；
- 动画导致对象位置不可追踪；
- 数据状态、快照或证据边界不可见；
- mock 数据没有标识；
- 视觉比例数字仅写在文档中、没有自动测试。


## 8. 当前原型自动化入口

- 信息工作区使用 `[data-information-workspace]` 标记。
- 图谱、ERD/血缘、模型图与滑杆工作台、领域范围图、开发状态、治理和操作时间线使用 `[data-visual-surface]` 标记。
- `scripts/validate_visual_coverage.py` 在 1440x900、1280x800、1024x768 三个视口计算矩形并集，不重复计算嵌套视觉区域。
- GitHub Actions 的 `visual-validation` Job 安装 Chromium 后执行该脚本。
- 该测试只验证视觉承载区域与页面错误，不替代真实用户可用性、帧率、LCP、CLS 和语义理解测试。
