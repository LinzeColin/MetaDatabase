## 20260629 - 10:50 CST / 20260629 - 12:50 AEST 邮件频率控制修复

- 目标：修复 Serenity 每个运行点重复发送相同 actionable 邮件的问题，确保只有“当前建议动作相对上一封已成功发送 actionable 邮件发生实质变化”时才发送。

- 规则：北京时间自然日最多发送 2 封 actionable 邮件；相同 signature 抑制为 `duplicate_action_signature`；同日第 3 封及以后抑制为 `daily_email_cap_reached`；维持/Info/无新增实质动作抑制为 `non_actionable`。
- 数据库：`notification_log` 新增 `action_signature`、`action_signature_hash`、`notification_kind`、`beijing_date`、`suppress_reason`、`related_run_id`、`created_at`，被抑制邮件也写入日志。

## 20260629 - 07:54 CST / 20260629 - 09:54 AEST 调度状态误报修复

- 目标：排查用户反馈的“为什么又不运行了”高频问题。
- 事实：LaunchAgent 正常加载并每 180 秒运行，最近 tick 持续写入 `automation_tick_log`，当前北京时间未到首个正式 slot `08:30`，所以 tick 正确返回 `no_due_slot`。
- 根因：本机 8765 app server 以 `--disable-autoscheduler` 启动，真实生产调度依赖外部 launchd；但 `/api/scheduler/status` 过去只读取 app 内置 scheduler 状态，导致页面/API 显示 `not_started`，容易被误判为整个自动化未运行。
- 修复：`app/core/application_server.py` 增加只读 launchd tick 聚合；最近 420 秒内有 `automation_tick_log` 记录时，状态接口返回 `status=success`、`scheduler_kind=launchd_interval`，同时保留 `application_server_autoscheduler_status=not_started` 解释内置 scheduler 未启用。
- 验证：`/opt/anaconda3/bin/python -m pytest -q tests/test_application_server.py` 为 16 passed；实时 `/api/scheduler/status` 返回 `status=success`、`scheduler_kind=launchd_interval`、最新 tick 为 `2026-06-29T07:53:20+08:00 no_due_slot`。
- 边界：未触发真实 run，未发送邮件，未启动或关闭 OpenD/MooMoo，未提交 runtime DB/WAL/lock。
