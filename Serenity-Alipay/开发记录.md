## 最近关键修复

### 20260703 - MooMoo/OpenD 系统安装发现与半导体暴跌复盘

- 目标：解释并修复用户已安装 MooMoo/OpenD 但 Serenity 生产运行没有使用 OpenD 的问题，同时记录“半导体暴跌但报告仍给出维持”的策略缺口。
- 事实：2026-07-02 的 R10 生产报告为 `degraded`，`MooMoo/moomoo_OpenD：不可用`；运行审计记录 `moomoo_OpenD socket not reachable at 127.0.0.1:11111`，并给出 `No moomoo_OpenD start entry found`。
- 根因：OpenD 自动发现逻辑只覆盖旧 workbench 和用户目录候选路径，没有发现新电脑实际安装位置 `/Applications/moomoo_OpenD.app`；即使后续能由自动化启动，默认清理策略也会关闭自动化自己启动的 OpenD，容易造成下一轮再次不可用。
- 修复：`moomoo_smoke` 增加 `/Applications` 系统应用目录发现，并过滤没有 OpenD/workbench 工件的空目录；`SERENITY_OPEND_KEEP_AUTO_STARTED` 默认改为保持自动启动的 OpenD 运行，避免 launch-close 循环。
- 验证：`tests/test_config.py`、`tests/test_moomoo_smoke.py`、`tests/test_moomoo_lifecycle.py`、`tests/test_moomoo_adapter.py`、`tests/test_pipeline_opend_lifecycle.py` 共 18 个测试通过；`moomoo-smoke --auto-start-opend --keep-auto-started-opend --require-ready` 通过，socket `127.0.0.1:11111` 可达；`preflight --json` 返回 `production_ready: true`。
- 策略缺口：该修复只保证 Serenity 可以使用 MooMoo/OpenD 数据，不等同于新增“半导体暴跌必须减仓”的交易规则。2026-07-02 报告中的半导体基金仍显示研究层“维持”，执行层因数据质量退化和执行锁为 `禁止新增 / 建议金额 0.00`；是否在板块暴跌时降权或减仓，需要单独实现半导体集中度、回撤、波动率和趋势阈值规则。
- 边界：没有改写 2026-07-02 历史报告、SQLite 历史记录、通知文件或旧运行快照；没有提交 runtime DB/WAL/lock/cache。

### 20260630 - 三基文件中文可读性修复

- 目标：把 GitHub main 上的三基文件彻底改成中文用户可读版本，避免再次出现“文件名中文、正文英文”的情况。
- 根因：旧三基文件由治理渲染器直接投影 canonical facts，字段名、功能说明、模型说明、公式说明和路线图任务大量保留英文。
- 修复：重写 `功能清单.md`、`开发记录.md`、`模型参数文件.md`，将功能、开发历史、模型、公式、参数和边界说明改为中文表达。
- 保护：增加中文可读性验证，允许保留 `FEAT-*`、`MOD-*`、`FORM-*`、`PARAM-*`、文件路径和命令，但禁止旧英文整句说明回流。
- 边界：不改评分算法、不改候选池、不改调仓逻辑、不改运行态 SQLite。

### 20260629 - 邮件频率控制修复

- 目标：修复每个运行点重复发送相同 actionable 邮件的问题。
- 根因：旧邮件策略只看本轮是否有非维持动作或紧急等级，没有比较上一封已成功发送的 actionable 邮件；`run_slot()` 和 `notify_run()` 又是两条可发送路径。
- 修复：新增 `action_signature` 和 `action_signature_hash`。签名只包含总体动作、Top5 顺序、基金代码和名称、每只基金动作、两位百分比目标权重和关键动作旗标。
- 邮件规则：相同动作结论不再重复发；北京时间同一天最多发 2 封 actionable 邮件；维持和普通信息默认不发。
- 数据库：`notification_log` 增加签名、通知类型、北京时间日期、抑制原因和关联 run。
- 验证：通知、集成、自动 tick、Serenity 优先级和 OpenD 生命周期相关测试共 22 个通过。
- 边界：没有改选基、评分、排序、目标权重、基金费用、OpenD/MooMoo 生命周期或交易行为。

### 20260629 - 调度状态误报修复

- 目标：修复用户看到“系统又不运行了”的误报。
- 事实：外部 LaunchAgent 正常每 180 秒 tick；当时北京时间还没到第一个真实运行点，所以 tick 正确记录为 `no_due_slot`。
- 根因：`/api/scheduler/status` 只显示 app 内置调度器状态；而当前生产调度实际由 launchd 执行，app 内置调度器是关闭的。
- 修复：状态接口增加 launchd tick 聚合。只要最近 420 秒内有 tick，就返回有效调度状态。
- 验证：`tests/test_application_server.py` 通过；接口返回 `success / launchd_interval`。
- 边界：没有触发真实 run、没有发邮件、没有启动或关闭 OpenD/MooMoo。

### 20260627 - 历史首次入池事实保护

- 目标：确认本机 app、GitHub main、benchmark/Alpha-Beta fail-closed 和历史完整性边界。
- 发现：`record_asset_pool_entries()` 可能在后续 run 中把基金首次进入持仓池的时间写成本轮时间。
- 修复：写入 `asset_pool_entry` 前，先从历史 `recommendation_snapshot + run_log` 找到该基金第一次进入候选池、持仓池或观察池的真实 run，再用 `INSERT OR IGNORE` 固化。
- 验证：应用服务、页面、基准、指标纪律和历史完整性测试共 36 个通过。
- 边界：没有运行会刷新 latest 审计文件的真实历史完整性 CLI，没有改写旧 SQLite 历史数据或旧报告。

### 20260624 - app 入口和人工复核高频问题

- 修复人工复核保存后待办不消失的问题：只要有效复核结果已写入 SQLite，即视为该复核待办已处理，后台刷新不再决定待办是否继续显示。
- 修复 app 打开两个 tab 的问题：入口只打开最终本地首页。
- 修复 Dock 图标持续跳动问题：launcher 不再等待 Python server，启动服务后打开首页并退出。
- 修复 OpenD/MooMoo 自动启动后被秒关的问题：用户已打开的进程绝不清理；自动化自己启动的进程只在任务完成且生命周期状态安全时清理。

## 不可回归规则

| 规则 | 必须怎么做 |
|---|---|
| 三基文件必须中文可读 | 用户说明、功能名、公式说明、任务说明必须用中文；代码标识符可以保留。 |
| 历史数据不可覆盖 | 旧 SQLite、旧报告、旧快照、创建时间、首次入池时间都不能被普通功能迭代重写。 |
| app 入口必须单 tab | 不得恢复 `downloads-entry.html` 双跳转或打开两个 tab 的行为。 |
| launcher 不得常驻跳动 | 不得让 `open-serenity` 等待 Python server。 |
| 人工复核保存即处理 | 复核结果写入数据库后，该待办必须立即从待处理列表消失。 |
| OpenD/MooMoo 生命周期要分清归属 | 用户打开的进程不清理；自动化自己启动的进程在任务完成后才按安全状态清理。 |
| OpenD 系统安装路径必须可发现 | 新电脑常见安装路径 `/Applications/moomoo_OpenD.app` 必须纳入 smoke/preflight/auto-start 发现范围。 |
| OpenD 定时运行不得启动后立即关闭 | 默认保持 Serenity 自动启动的 OpenD 运行，避免每日调度窗口前后反复 launch-close 导致 socket 不可用。 |
| 半导体暴跌减仓不能靠事后解释 | 如需暴跌后自动降权或减仓，必须有明确板块回撤、集中度、波动率和趋势阈值规则及回放验证。 |
| 邮件只发重要变化 | 维持、重复结论、普通数据不足不应反复发邮件。 |
| 不提交运行态文件到 main | `serenity_daily.sqlite`、WAL、SHM、lock、cache 属于本机运行态。 |

## 后续推荐动作

1. 每次改三基文件后，运行中文可读性验证。
2. 每次改邮件、调度、复核、入口、OpenD/MooMoo 生命周期后，先跑对应回归测试。
3. 新电脑迁移时，先 clone GitHub main，再单独恢复 runtime 备份，不要把运行态数据混进源码提交。
